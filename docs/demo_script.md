## 3 · Architecture (2:00 – 3:15)

**Screen**: open `presentation.pdf` slide 3 (the architecture diagram).

**Script:**

> "Under the hood, six pipeline stages." *(point to each as you name it)* "**Ingest** — we parse SP 21 with PyMuPDF on the 'SUMMARY OF / IS XXXX : YYYY' header pattern, extracting 559 structured records.
> **Index** — we build two indices in parallel: bge-m3 dense embeddings in FAISS, and BM25 sparse over the same chunks.
> **Fuse** — Reciprocal Rank Fusion merges both ranked lists with a parameter-free formula.
> **Rerank** — bge-reranker-v2-m3 cross-encoder re-scores the top 25 with full query-passage attention.
> **Guard** — every IS code is checked against our whitelist before it leaves the system.
> **Serve** — `inference.py` for the judges, FastAPI plus Next.js for the demo. Critically, the eval path makes zero external API calls — Gemini is UI polish only."

## 4 · Retrieval strategy (3:15 – 4:15)

**Screen**: slide 4 (chunking + 4-component cascade).

**Script:**

> "Why hybrid retrieval? Each component covers a failure mode the others miss.
> **BM25** handles rare technical tokens — grade codes like M30, OPC33, exact terms like *mortice* — where pure semantic similarity fails.
> **bge-m3 dense** handles paraphrase. An MSE owner who writes 'we make blocks for walls that aren't load-bearing' should still find IS 2185 Part 2 — even though those exact words never appear in the standard.
> **RRF** fuses them without any learned weighting — robust across query distributions.
> **The cross-encoder** then does a final pass with explicit attention between query and each candidate's title plus scope. That's where most of the rank-1 wins come from."
